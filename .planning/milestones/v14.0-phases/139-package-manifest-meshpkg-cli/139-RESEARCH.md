## User Constraints (from CONTEXT.md)

### Locked Decisions

**Manifest dependency syntax:**
- Both shorthand (`foo = "1.0.0"`) and table form (`foo = { version = "1.0.0" }`) are supported
- Caret semver only for version constraints: `^1.0` means `>=1.0.0 <2.0.0`
- Table form supports all three source types: registry (`version`), local path (`path = "../foo"`), and git (`git = "https://..."`)
- `[package]` section: `name` and `version` are required; `description`, `license`, `authors` are optional
- Required fields are enforced at publish time, not at local development time

**CLI output & UX:**
- Spinner with status text during `meshpkg install` and `meshpkg publish` (e.g., "Downloading foo@1.0.0...")
- `meshpkg search` outputs an aligned table with columns: name, version, description
- Success/error output uses colored check/cross with message (e.g., `✓ Published foo@1.0.0`)
- `--json` flag available on all commands for machine-readable output

**mesh.lock format:**
- TOML format (consistent with mesh.toml)
- Each locked entry contains: name, version, sha256, source URL
- mesh.lock should be committed to git for reproducible builds
- `meshpkg install` with existing lock uses exact pins; without a lock, resolves and generates one

**Package install layout:**
- Packages install to `.mesh/packages/<name>@<version>/` (project-local, not global)
- Published tarball structure: `mesh.toml` at root + `src/` directory at root
- Compiler discovers installed packages by reading `mesh.lock` to find `.mesh/packages/` paths
- `.mesh/packages/` is gitignored by default (mesh.lock provides reproducibility)

## Summary

Phase 139 builds on the existing `mesh-pkg` crate (which already handles manifest parsing, git dependencies, path dependencies, lockfile I/O, and scaffolding) by adding two missing pieces: (1) a `Dependency::Registry` variant with shorthand string syntax and a `version` field in table form, and (2) a new standalone `meshpkg` CLI binary crate that wraps everything with publish/install/search/login subcommands.

The existing `mesh-pkg` crate already has `toml = "0.8"`, `serde`, `serde_json`, and `semver` as workspace dependencies. The workspace already has `ureq = "3"` (with `gzip` feature) in `mesh-rt` and `sha2 = "0.10"` — these must be added as workspace-level dependencies for the new `meshpkg` crate to share them. New additions needed: `tar = "0.4"`, `flate2 = "1"`, `indicatif = "0.17"`, `colored = "2"`, and optionally `dirs = "5"` for `~/.mesh/credentials` path resolution.

The critical design insight: the current `Lockfile` / `LockedPackage` structs must be extended — `LockedPackage` needs a `sha256` field and the `source` field should hold the registry download URL (not just the package URL). The `meshpkg install` command must handle two modes: (a) with existing `mesh.lock` — use exact pins to download; (b) without `mesh.lock` — query registry for versions matching manifest constraints, then generate lock.

**Primary recommendation:** Create `compiler/meshpkg/` as a new binary crate mirroring how `meshc` uses `mesh-pkg`. Extend `mesh-pkg` first (Plan 01), then build the CLI binary (Plan 02). Use `ureq 3` (already a workspace dep), `indicatif 0.17` for spinners, `colored 2` for success/error output, `tar + flate2` for tarball operations.

### Pattern 1: Registry Dependency Variant (shorthand + table form)

The existing `Dependency` enum uses `#[serde(untagged)]` — the new `Registry` variant must be ordered FIRST in the enum so TOML strings (`foo = "1.0.0"`) and version-only tables (`foo = { version = "1.0.0" }`) get matched before `Git` and `Path` variants.

**What:** Extend manifest.rs with a `Registry` variant that handles both shorthand string (`"1.0.0"`) and table form (`{ version = "1.0.0" }`).
**When to use:** When the Dependency is specified with only a `version` field (registry source) in mesh.toml.

```rust
// Source: adapted from existing manifest.rs pattern + TOML serde untagged rules
#[derive(Debug, Clone, Deserialize)]
#[serde(untagged)]
pub enum Dependency {
    // Must be first: bare string "1.0.0" matches this
    RegistryShorthand(String),
    // Table with only version key: { version = "1.0.0" }
    Registry {
        version: String,
    },
    Git {
        git: String,
        #[serde(default)]
        rev: Option<String>,
        #[serde(default)]
        branch: Option<String>,
        #[serde(default)]
        tag: Option<String>,
    },
    Path {
        path: String,
    },
}

impl Dependency {
    /// Normalize to canonical form — shorthand "1.0.0" becomes Registry { version }.
    pub fn version(&self) -> Option<&str> {
        match self {
            Dependency::RegistryShorthand(v) => Some(v),
            Dependency::Registry { version } => Some(version),
            _ => None,
        }
    }
}
```

### Pattern 2: Extended Lockfile with SHA-256 and Source URL

The existing `LockedPackage` only has `name`, `source`, `revision`. For registry packages, we need `sha256` and `source_url` (the download URL). The lockfile version should bump to 2 to signal the new format — or keep version 1 and make new fields optional (recommended to avoid breaking existing lockfiles).

```rust
// Source: extending existing lockfile.rs
#[derive(Debug, Serialize, Deserialize, PartialEq, Clone)]
pub struct LockedPackage {
    pub name: String,
    pub version: String,           // NEW: explicit version string
    pub source: String,            // registry URL, git URL, or path
    pub revision: String,          // git SHA, "local", or version for registry
    #[serde(default)]
    pub sha256: Option<String>,    // NEW: hex-encoded SHA-256 of tarball
}
```

### Pattern 3: Tarball Creation for Publish

**What:** Package the project as `<name>-<version>.tar.gz` containing `mesh.toml` at root and `src/` directory.
**When to use:** `meshpkg publish` command.

```rust
// Source: Rust Cookbook (rust-lang-nursery.github.io/rust-cookbook/compression/tar.html)
use flate2::write::GzEncoder;
use flate2::Compression;
use tar::Builder;
use sha2::{Sha256, Digest};

fn create_tarball(project_dir: &Path, manifest: &Manifest) -> Result<(Vec<u8>, String), String> {
    let mut buf = Vec::new();
    {
        let enc = GzEncoder::new(&mut buf, Compression::default());
        let mut archive = Builder::new(enc);
        // Add mesh.toml at root
        archive.append_path_with_name(project_dir.join("mesh.toml"), "mesh.toml")
            .map_err(|e| format!("Failed to add mesh.toml: {}", e))?;
        // Add src/ directory
        let src_dir = project_dir.join("src");
        if src_dir.exists() {
            archive.append_dir_all("src", &src_dir)
                .map_err(|e| format!("Failed to add src/: {}", e))?;
        }
        archive.finish().map_err(|e| format!("Failed to finalize archive: {}", e))?;
    }
    // Compute SHA-256 of tarball bytes
    let hash = hex::encode(Sha256::digest(&buf));
    Ok((buf, hash))
}
```

Note: `hex` encoding — `sha2` returns bytes, can use `format!("{:x}", ...)` with Display or add `hex = "0.4"` crate. Since `mesh-rt` already has `sha2` and we can use `format!("{:02x}", byte)` in a loop, or use the `sha2` crate's hex output. Check: `sha2` 0.10 uses `digest` traits — use `Sha256::digest()` returning `GenericArray`, then iterate bytes or use `base16ct` crate. Simplest: `format!("{:x}", digest)` works because `GenericArray` implements `LowerHex`. This is verified in the existing `mesh-rt` code that does SHA-256 for WebSocket.

### Pattern 4: HTTP Publish with ureq 3

**What:** Upload tarball bytes to registry via HTTP POST.
**When to use:** `meshpkg publish` after tarball creation.

```rust
// Source: ureq 3 API (docs.rs/ureq/latest/ureq)
use ureq::Agent;

fn publish_to_registry(
    tarball: &[u8],
    sha256: &str,
    name: &str,
    version: &str,
    token: &str,
    registry_url: &str,
) -> Result<(), String> {
    let agent = Agent::new_with_defaults();
    let url = format!("{}/api/v1/packages", registry_url);
    let response = agent
        .post(&url)
        .header("Authorization", &format!("Bearer {}", token))
        .header("Content-Type", "application/octet-stream")
        .header("X-Package-Name", name)
        .header("X-Package-Version", version)
        .header("X-Package-SHA256", sha256)
        .send(tarball)
        .map_err(|e| format!("Failed to publish: {}", e))?;
    // ureq 3: check status via response.status()
    if response.status() == 200 || response.status() == 201 {
        Ok(())
    } else {
        Err(format!("Registry rejected publish: HTTP {}", response.status()))
    }
}
```

Note: ureq 3 changed `.set()` to `.header()`, `.send_bytes()` to `.send()`, and response body reading to `.body_mut().read_to_string()`. This is confirmed by existing mesh-rt/http/client.rs usage.

### Pattern 5: Spinner with indicatif

**What:** Show animated spinner during long-running operations.
**When to use:** All network operations (download, upload, search).

```rust
// Source: indicatif docs (docs.rs/indicatif)
use indicatif::{ProgressBar, ProgressStyle};
use std::time::Duration;

fn with_spinner<T, F: FnOnce() -> Result<T, String>>(
    msg: &str,
    f: F,
) -> Result<T, String> {
    let pb = ProgressBar::new_spinner();
    pb.set_style(
        ProgressStyle::with_template("{spinner:.cyan} {msg}")
            .unwrap()
            .tick_strings(&["⠋", "⠙", "⠹", "⠸", "⠼", "⠴", "⠦", "⠧", "⠇", "⠏", ""]),
    );
    pb.set_message(msg.to_string());
    pb.enable_steady_tick(Duration::from_millis(80));
    let result = f();
    pb.finish_and_clear();
    result
}
```

Note: When `--json` flag is active, spinner must be suppressed. Check `!json_mode` before creating the spinner. `indicatif` auto-hides on non-TTY (e.g., pipes) so piping to files works correctly.

### Pattern 6: Credentials File (~/.mesh/credentials)

**What:** Store auth token for the registry in a TOML file.
**When to use:** `meshpkg login` writes; `meshpkg publish` reads.

```rust
// Source: pattern derived from Cargo's credentials.toml approach
use std::path::PathBuf;

fn credentials_path() -> PathBuf {
    dirs::home_dir()
        .unwrap_or_else(|| PathBuf::from("."))
        .join(".mesh")
        .join("credentials")
}

// File format (TOML):
// [registry]
// token = "abc123..."

fn read_token() -> Result<String, String> {
    let path = credentials_path();
    let content = std::fs::read_to_string(&path)
        .map_err(|_| "Not logged in. Run `meshpkg login` first.".to_string())?;
    let table: toml::Table = toml::from_str(&content)
        .map_err(|e| format!("Corrupted credentials file: {}", e))?;
    table.get("registry")
        .and_then(|r| r.get("token"))
        .and_then(|t| t.as_str())
        .map(|s| s.to_string())
        .ok_or_else(|| "No token found in credentials file.".to_string())
}

fn write_token(token: &str) -> Result<(), String> {
    let path = credentials_path();
    std::fs::create_dir_all(path.parent().unwrap())
        .map_err(|e| format!("Failed to create ~/.mesh/: {}", e))?;
    let content = format!("[registry]\ntoken = \"{}\"\n", token);
    std::fs::write(&path, content)
        .map_err(|e| format!("Failed to write credentials: {}", e))
}
```

    let max_name = results.iter().map(|r| r.name.len()).max().unwrap_or(4);
    let max_version = results.iter().map(|r| r.version.len()).max().unwrap_or(7);

    }

### Anti-Patterns to Avoid

- **Ordering `Registry` variant after `Git`/`Path` in untagged enum:** TOML's untagged deserialization tries variants in order. `RegistryShorthand(String)` will match ANY string including "malformed" if placed first — but since bare `path = "..."` requires a key, ordering `RegistryShorthand` before `Git` and `Path` is correct as a bare string cannot match a table form variant.
- **Blocking inside spinner's steady tick thread:** `indicatif`'s `enable_steady_tick` spawns its own thread. Don't hold locks that the steady tick thread needs.
- **Using `~` directly in path strings:** Shell expansion does not happen in Rust. Always use `dirs::home_dir()` or read `$HOME`.
- **Hardcoding registry URL:** The registry URL should be configurable — read from `mesh.toml [registries]` section or a default constant (Phase 140 defines the actual URL). Use a const default or an environment variable `MESH_REGISTRY_URL`.
- **Not verifying SHA-256 on install:** Always verify the downloaded tarball's SHA-256 against the lockfile entry before extracting. Missing this is a supply-chain attack vector.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Tarball creation/extraction | Custom file bundling loop | `tar` + `flate2` crates | Edge cases: symlinks, permissions, directory ordering, POSIX header format |
| Terminal spinners | Manual `\r` cursor tricks | `indicatif` | Thread safety, TTY detection, proper cleanup on panic/Ctrl-C |
| Home directory path | `$HOME` env var read | `dirs::home_dir()` | `$HOME` unset on some systems; `dirs` handles Windows, macOS, Linux correctly |
| SHA-256 hex formatting | Manual byte-to-hex loop | `sha2` with `format!("{:02x}", byte)` or `hex` crate | Subtle bugs in hex encoding; `sha2` already in workspace |
| TOML table/string deserialization | Custom Deserialize impls | `serde` untagged enum | Correctly handles both `"1.0.0"` and `{ version = "1.0.0" }` with zero extra code |

**Key insight:** The tarball format has many subtle rules (directory entries before file entries, correct mode bits, proper path handling). `tar::Builder` handles all of this; hand-rolling creates interoperability problems with standard `tar` tools.

## Common Pitfalls

### Pitfall 1: Untagged Enum Variant Order

**What goes wrong:** `Registry { version }` table form gets matched as `Path { path }` or `Git { git }` if variant ordering is wrong, causing a confusing parse error.
**Why it happens:** `serde(untagged)` tries variants in declaration order. A `{ version = "1.0.0" }` TOML table could hypothetically match any struct variant with optional fields.
**How to avoid:** Declare `RegistryShorthand(String)` first (catches bare strings), then `Registry { version }` (catches `{ version = "..." }` tables), then `Git` (requires `git` key), then `Path` (requires `path` key). Write tests for all four variants.
**Warning signs:** `parse_full_manifest` tests failing with "expected registry dependency, got path" or similar.

### Pitfall 2: ureq 3 API vs ureq 2 API Mismatch

**What goes wrong:** Using `.set()`, `.send_bytes()`, `.into_string()` — these are ureq 2 APIs. ureq 3 uses `.header()`, `.send()`, `.body_mut().read_to_string()`.
**Why it happens:** ureq 3 is a breaking-change rewrite of the API. Documentation and examples online often mix versions.
**How to avoid:** Always check existing `mesh-rt/src/http/client.rs` — it correctly uses ureq 3. Use the same patterns: `agent.post(url).header(k, v).send(bytes)` for upload, `response.body_mut().read_to_string()` for response.
**Warning signs:** Compile errors on `.set()` method not found, or `send_bytes` not found.

### Pitfall 3: Lockfile Field Migration

**What goes wrong:** Adding `sha256` to `LockedPackage` breaks deserialization of existing lockfiles that don't have this field.
**Why it happens:** Serde's `Deserialize` for structs fails on unknown or missing required fields by default.
**How to avoid:** Mark new fields with `#[serde(default)]` — `Option<String>` for `sha256` with `#[serde(default)]` means old lockfiles (without sha256) still parse successfully. Non-registry packages (git, path) can have `sha256: None`.
**Warning signs:** Unit tests for `lockfile_round_trip` failing after adding new fields.

### Pitfall 4: SHA-256 Verification Race Condition

**What goes wrong:** Tarball is extracted before SHA-256 verification completes, leaving partial state on disk if verification fails.
**Why it happens:** Streaming extraction skips buffering the whole file.
**How to avoid:** Buffer the entire tarball into a `Vec<u8>` first, verify SHA-256, then extract from the buffer. For packages up to a few MB this is fine. Use `let mut buf = Vec::new(); response.body_mut().read_to_end(&mut buf)` then verify before extracting.
**Warning signs:** Corrupted package directories when downloads are interrupted.

### Pitfall 5: Missing `dirs` in Workspace

**What goes wrong:** `dirs::home_dir()` returns `None` on unusual systems. Alternatively, `$HOME` is set but points to a read-only directory.
**Why it happens:** `dirs` is not a current workspace dependency.
**How to avoid:** Add `dirs = "5"` to workspace dependencies. Always handle `None` from `home_dir()` with a fallback to `PathBuf::from(".")` and a clear error message. Create `~/.mesh/` directory if it doesn't exist before writing credentials.
**Warning signs:** Crash with unwrap on `home_dir()` on CI systems without `$HOME` set.

### Pitfall 6: Tarball Path Stripping

**What goes wrong:** Extracted package ends up at `.mesh/packages/foo@1.0.0/foo-1.0.0/src/` (extra nesting) instead of `.mesh/packages/foo@1.0.0/src/`.
**Why it happens:** The tarball was created with a top-level directory prefix, or `archive.unpack()` preserves the archive's root prefix.
**How to avoid:** When creating the tarball for publish, do NOT add a top-level directory — add `mesh.toml` and `src/` directly at the archive root. When extracting on install, extract directly to `.mesh/packages/<name>@<version>/`. The `tar::Archive::unpack()` call handles this if the archive was created without a prefix.
**Warning signs:** `src/` directory not found after install because it's nested one level deeper.

fn write_token(token: &str) -> Result<(), String> {

        .unwrap_or_else(|| std::path::PathBuf::from("."))

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| `ureq 2` `.set()` / `.send_bytes()` | `ureq 3` `.header()` / `.send()` | ureq 3.0 (2024) | API fully incompatible; meshc already uses ureq 3 correctly |
| `Dependency` without registry support | Add `Registry` / `RegistryShorthand` variants | Phase 139 | Enables `version = "1.0.0"` in mesh.toml |
| `LockedPackage` without sha256 | Add optional `sha256` and explicit `version` | Phase 139 | Enables tamper-verified installs |
| `meshc deps` for all dependency management | `meshpkg` standalone binary | Phase 139 | Separate concerns: compiler vs package manager |

**What currently exists in mesh-pkg:**
- `Dependency::Git` and `Dependency::Path` — works, tested
- `Lockfile` / `LockedPackage` — works, tested (24 tests passing)
- `resolve_dependencies()` — git + path only, no registry
- `scaffold_project()` — creates mesh.toml + main.mpl

**What is missing (Phase 139 adds):**
- `Dependency::Registry` variant (shorthand string + table form)
- `license` field in `Package` struct

## Open Questions

1. **Registry URL constant / configuration**
   - What we know: Phase 140 defines the registry server (hosted at a real URL); Phase 139 is client-only
   - What's unclear: Should the registry URL be hardcoded as a const (`https://registry.meshlang.dev`), configurable via `mesh.toml [registries]` section, or an env var?
   - Recommendation: Use a default const `MESH_REGISTRY_URL = "https://registry.meshlang.dev"` and support `--registry` flag override on publish/install commands. The `[registries]` section in mesh.toml can be a Phase 141+ feature.

2. **Registry API contract (search/install endpoints)**
   - What we know: Phase 140 defines the actual API; Phase 139 must call it
   - What's unclear: The exact JSON schema for search results (`/api/v1/packages/search?q=...`) and the download URL format
   - Recommendation: Define the API contract in Plan 01's comments as a forward-declaration. Use a simple schema: `GET /api/v1/packages?search=<query>` returns `[{ name, version, description }]` and `GET /api/v1/packages/<name>/<version>/download` returns the tarball bytes. Phase 140 must implement exactly this schema.

3. **`meshpkg install` with no arguments (install all from mesh.toml)**
   - What we know: Context says `meshpkg install <name>` installs a named package
   - What's unclear: Does `meshpkg install` with no argument also install all dependencies from mesh.toml? (Like `npm install` vs `npm install <pkg>`)
   - Recommendation: Support both modes: `meshpkg install` (no arg) reads mesh.toml and installs all dependencies into `.mesh/packages/`; `meshpkg install <name>` adds a dependency to mesh.toml and installs it. This matches the success criteria "meshpkg install generates mesh.lock pinning exact versions".

4. **Token input for `meshpkg login`**
   - What we know: Login stores token in `~/.mesh/credentials`
   - What's unclear: Is the token read from `--token` flag, or from stdin interactively?
   - Recommendation: Support both: `--token <value>` for scripting; interactive stdin prompt (using `rpassword` or simple `read_line`) when no flag given. `rpassword` is not in workspace — use `read_line` for simplicity since this is a CLI tool not a web app.
