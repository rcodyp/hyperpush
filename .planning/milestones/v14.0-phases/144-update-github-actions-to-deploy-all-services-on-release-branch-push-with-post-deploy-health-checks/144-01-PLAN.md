  <action>
Create a new GitHub Actions workflow file `.github/workflows/deploy-services.yml` with the following exact structure:

**Trigger:** `push: tags: ['v*']` and `workflow_dispatch` (mirrors release.yml pattern; fires on tag push, also dispatchable manually for testing).

**Concurrency:** `group: deploy-fly-${{ github.ref_name }}` with `cancel-in-progress: false` — prevents parallel deploys of the same tag, never cancels an in-flight deploy.

**Three jobs:**

1. `deploy-registry` — deploys `mesh-registry` to Fly.io:
   - `runs-on: ubuntu-latest`
   - steps: `actions/checkout@v4`, `superfly/flyctl-actions/setup-flyctl@master`, then run `flyctl deploy --remote-only` with `working-directory: registry` and `env: FLY_API_TOKEN: ${{ secrets.FLY_API_TOKEN }}`

2. `deploy-packages-website` — deploys `mesh-packages` to Fly.io:
   - `runs-on: ubuntu-latest`
   - steps: `actions/checkout@v4`, `superfly/flyctl-actions/setup-flyctl@master`, then run `flyctl deploy --remote-only` with `working-directory: packages-website` and `env: FLY_API_TOKEN: ${{ secrets.FLY_API_TOKEN }}`

3. `health-check` — polls all three endpoints after both deploys succeed:
   - `needs: [deploy-registry, deploy-packages-website]`
   - `runs-on: ubuntu-latest`
   - Three inline curl steps (no marketplace action — zero external dependency):
     - Check registry API: `curl -sf --retry 5 --retry-delay 10 --retry-connrefused https://api.packages.meshlang.dev/api/v1/packages > /dev/null && echo "Registry API: OK"`
     - Check packages website: `curl -sf --retry 5 --retry-delay 10 --retry-connrefused https://packages.meshlang.dev > /dev/null && echo "Packages website: OK"`
     - Check docs site: `curl -sf --retry 3 --retry-delay 5 https://meshlang.dev > /dev/null && echo "Docs site: OK"`

Use descriptive `name:` fields for each job and step so the Actions UI is readable. No permissions block needed (no write operations in this workflow).
