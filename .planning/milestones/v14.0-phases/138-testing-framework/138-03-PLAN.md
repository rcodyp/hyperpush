<objective>
Wire the test DSL (test/describe/setup/teardown/assert_receive/Test.mock_actor) through the
MIR lowerer and add e2e test fixtures.

This plan has two main concerns:
1. **DSL lowering**: Detect `.test.mpl` files during compilation and transform `test`/`describe`/
   `setup`/`teardown` blocks into a generated `main()` harness that calls `mesh_test_begin`,
   runs test bodies via catch_unwind, calls `mesh_test_pass`/`mesh_test_fail_msg`, then
   `mesh_test_summary`.
2. **Mock actors + assert_receive**: Add `Test.mock_actor` type to infer.rs, add
   `mesh_test_mock_actor` runtime function, wire through 5-point registration (including the
   LLVM declaration in intrinsics.rs), and lower `assert_receive pattern, timeout` into
   `mesh_actor_receive(timeout) + pattern match`.

Purpose: This is where the test framework becomes usable end-to-end.
Output: Updated `lower.rs` + `intrinsics.rs` + `infer.rs` + `test.rs`, three `.test.mpl` e2e fixture files.
</objective>

    **Detection of test mode:**
    At the start of `lower_to_mir_raw()` (or in a preprocessing step), scan the top-level items
    in the parsed AST for function calls named `test` or `describe`. If any are found, set
    `is_test_mode = true` and generate the test harness.

    If lower_to_mir_raw has access to the Parse tree, look for top-level ExprStmt nodes

    **Test harness generation strategy:**

    The simplest correct approach: transform test/describe/setup/teardown calls into a
    generated `main()` function at the MIR level.

    Before generating MIR for the file's normal content:
    1. Collect all top-level `test "desc" do body end` calls — these become individual test entries.
    2. Collect all `describe "group" do ... end` calls — inside each, collect nested `test`,
       `setup`, and `teardown` blocks.
    3. Generate a `main()` MIR function body that:
       - Calls `mesh_rt_init_actor(1)` to start the scheduler for mock actors
       - Records start time via `std::time::Instant::now()` (or use a runtime timer call)
       - For each test (in order, respecting describe grouping):
         a. Calls `mesh_test_cleanup_actors()` before the test body
         b. If in a describe with setup: runs the setup body
         c. Calls `mesh_test_begin(test_label_ptr)` where label is `"desc"` or `"Group > desc"`
         d. Wraps the test body in a Rust `std::panic::catch_unwind` equivalent:
            - In MIR: emit the test body instructions; if mesh_panic is called, it will unwind
            - Use existing `catch_unwind` infrastructure used by actor crash isolation
         e. On success: calls `mesh_test_pass()`
         f. On panic: calls `mesh_test_fail_msg(error_msg_ptr)`
         g. If in a describe with teardown: runs the teardown body
       - After all tests: calls `mesh_test_summary(pass_count, fail_count, elapsed_ms)`
       - Exits with `process::exit(if fail_count > 0 { 1 } else { 0 })`

    **Implementation notes for the MIR generation:**

    The Mesh lowerer already generates MIR `main` functions. In test mode, instead of lowering
    the file's top-level expressions as the main body, generate the harness. The original
    test/describe call AST nodes are consumed by the harness generator, not lowered normally.

    For `assert expr`:
    - Lower the expression `expr` to get a boolean value
    - Emit: `call mesh_test_assert(bool_val, expr_src_str_ptr, file_ptr, file_len, line)`
    - `expr_src_str_ptr`: allocate a MeshString from the source span text (use `mesh_string_new`)
    - `file_ptr` / `file_len`: use the file path string
    - `line`: use the line number from the source span

    For `assert_eq a, b`:
    - Lower `a` and `b`
    - Type-dispatch: convert each to a MeshString using `mesh_int_to_string`, `mesh_bool_to_string`,
      `mesh_float_to_string`, or use the value directly if it's already a `*const MeshString`
    - Emit: `call mesh_test_assert_eq(lhs_str, rhs_str, expr_src_ptr, file_ptr, file_len, line)`

    For `assert_ne a, b`: symmetric to assert_eq

    For `assert_raises fn() do body end`:
    - The fn body is a no-arg closure
    - Lower to: `call mesh_test_assert_raises(fn_ptr, env_ptr, file_ptr, file_len, line)`

    For `assert_receive pattern, timeout_ms`:
    - Lower to:
      ```
      let __msg_ptr = call mesh_actor_receive(timeout_ms_val)
      if __msg_ptr == null:
        call mesh_test_fail_msg("assert_receive timed out waiting for pattern after {timeout_ms}ms")
        call mesh_panic(...)  # unwind
      else:
        match *__msg_ptr with pattern -> () | _ ->
          call mesh_test_fail_msg("assert_receive: received unexpected message")
          call mesh_panic(...)
      ```
    - When timeout_ms is omitted (just `assert_receive pattern`), use 100ms as default.

    **Practical implementation approach:**
    If the full harness generation is complex to implement in one pass, a simpler approach:
    Generate a Mesh source string programmatically that wraps the test bodies, then parse
    and lower that generated source. This avoids direct MIR construction complexity.

    However, the recommended approach is: Recognize test/describe/setup/teardown/assert/assert_eq/
    assert_ne/assert_raises/assert_receive as special call-like constructs during lowering.
    When `is_test_mode`, these are handled by the lowerer's special-case code rather than
    dispatched as normal function calls.

    **Critical constraint from research (do not violate):**
    - `test`, `describe`, `setup`, `teardown` are NOT new parser keywords — they are function-like
      calls that the lowerer recognizes in test mode
    - Test builtins are only available in test mode — never visible in regular .mpl builds
    - Each test binary exits with code 0 (all pass) or 1 (any fail)

    **assert_receive lowering uses:**
    - `mesh_actor_receive(timeout_ms: i64) -> *const u8` — already in known_functions
    - Null check on the return value → timeout failure path
    - Pattern match on non-null → success or mismatch failure path
