**Multi-line pipe syntax (from Phase 126):**
```mpl
# Single-line (before):
let r = HTTP.router() |> HTTP.on_post("/path", handler) |> HTTP.on_get("/path2", handler2)

# Multi-line (after) — |> at start of continuation line:
let r = HTTP.router()
  |> HTTP.on_post("/path", handler)
  |> HTTP.on_get("/path2", handler2)
```
Both forms are semantically identical. The parser accepts `|>` and `|N>` at either end of a line.

**Type alias syntax (from Phase 127):**
```mpl
# In types/event.mpl:
pub type Fingerprint = String

# In fingerprint.mpl (after import):
from Types.Event import ..., Fingerprint
fn normalize_message(msg :: String) -> Fingerprint do ... end
fn fingerprint_from_frames(frames, msg :: String) -> Fingerprint do ... end
pub fn compute_fingerprint(payload :: EventPayload) -> Fingerprint do ... end
```
Type aliases are transparent to the type checker — `Fingerprint` and `String` unify directly.

**Current mesher/main.mpl router chain (lines 120-156):**
The entire router setup is a single expression inside `HTTP.serve(...)`:
```mpl
HTTP.serve((HTTP.router()
  |> HTTP.on_post("/api/v1/events", handle_event)
  |> HTTP.on_post("/api/v1/events/bulk", handle_bulk)
  ... (37 routes total) ...
  |> HTTP.on_get("/api/v1/projects/:project_id/storage", handle_get_project_storage)), http_port)
```
This is already broken across lines with `|>` continuation — but all on the same indentation level inside one large expression. The task is to verify this is already valid multi-line syntax OR to extract it to a `let router = ...` binding with proper multi-line formatting.
