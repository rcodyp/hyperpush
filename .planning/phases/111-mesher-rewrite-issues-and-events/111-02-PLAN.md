  <action>
Rewrite 2 complex issue queries:

1. **upsert_issue**: Replace the raw `INSERT INTO issues ... ON CONFLICT ... DO UPDATE SET ... RETURNING id::text` with the ORM upsert API:
   - Build fields map: `%{"project_id" => project_id, "fingerprint" => fingerprint, "title" => title, "level" => level, "event_count" => "1"}`
   - Call: `Repo.insert_or_update(pool, Issue.__table__(), fields, ["project_id", "fingerprint"], ["event_count", "last_seen", "status"])`
   - The Repo.insert_or_update API (from Phase 109) generates `INSERT ... ON CONFLICT (conflict_targets) DO UPDATE SET ... RETURNING *`
   - IMPORTANT: The current raw SQL has custom SET expressions: `event_count = issues.event_count + 1`, `last_seen = now()`, `status = CASE WHEN ... END`. The Repo.insert_or_update API uses `SET field = EXCLUDED.field` by default, which does NOT support `event_count + 1` or conditional CASE expressions.
   - DECISION: Since Repo.insert_or_update cannot express `event_count + 1` or `CASE WHEN status = 'resolved' THEN 'unresolved' ELSE status END`, this query MUST retain Repo.query_raw. Rewrite it to use the cleanest possible Repo.query_raw call with clear documentation explaining WHY the ORM upsert cannot be used (custom SET expressions with arithmetic and conditionals).
   - Keep the existing Repo.query_raw implementation but add a comment: `# ORM upsert cannot express event_count + 1 or CASE status logic -- intentional raw SQL`

2. **check_volume_spikes**: This is an extremely complex UPDATE with nested subqueries, joins, HAVING, GREATEST, and interval arithmetic. The ORM cannot express this level of complexity.
   - Keep as Repo.execute_raw but add a comment: `# ORM cannot express nested subquery with JOIN + HAVING + GREATEST + interval arithmetic -- intentional raw SQL`
   - This is acceptable per the phase success criteria: "All 12 issue + event queries replaced with ORM calls" -- these 2 are the legitimate exceptions where raw SQL is the correct approach due to SQL complexity. The ORM does not have the expressiveness for these patterns.

For both functions, preserve exact signatures.

<task type="auto">
  <name>Task 2: Rewrite insert_event and extract_event_fields with fragments</name>
  <files>mesher/storage/writer.mpl, mesher/storage/queries.mpl</files>
  <action>
Rewrite 2 event queries to use ORM fragments where feasible:

1. **insert_event** (in writer.mpl): The current INSERT uses `SELECT $1::uuid, $2::uuid, j->>'level', j->>'message', $3, (j->'exception')::jsonb, ... FROM (SELECT $4::jsonb AS j) AS sub`. This is a SELECT-based INSERT that extracts JSONB fields server-side.
   - This query pattern does NOT fit Repo.insert (which takes a Map<String,String> of field->value) because the values are JSONB extraction expressions, not literal strings.
   - APPROACH: Keep as Repo.execute_raw but restructure to use clear JSONB fragment annotations in comments. The core issue is that Repo.insert cannot express `j->>'level'` as a value -- it only accepts literal strings.
   - Add comment: `# Repo.insert cannot express server-side JSONB extraction (j->>'field') -- intentional raw SQL for INSERT ... SELECT pattern`
   - Alternative: Could restructure to extract fields client-side in Mesh and then use Repo.insert with a Map. However, this would require JSON parsing in Mesh (from_json on the event JSON), which is blocked by the cross-module from_json limitation (decision [88-02]). Document this in the comment.

2. **extract_event_fields** (in queries.mpl): This is an extremely complex JSONB computation query with CASE/WHEN, jsonb_array_elements, string_agg, COALESCE, and conditional fingerprint fallback chains. The ORM cannot express any part of this.
   - Keep as Repo.query_raw but add comment: `# Complex JSONB fingerprint computation -- ORM fragments cannot express CASE/jsonb_array_elements/string_agg chain`
   - This is the single most complex SQL query in Mesher and is intentionally raw.

For both functions, preserve exact signatures. These queries represent the legitimate boundary of ORM expressiveness -- they require server-side JSONB computation that the fragment API was not designed to compose.
