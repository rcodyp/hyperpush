**Step 1 — Reproduce the bug with a minimal Mesh program:**

Create a test in `crates/meshc/tests/e2e.rs` that compiles a Mesh program containing:

```mesh
fn run() -> Int!String do
  let db = Sqlite.open(":memory:")?
  let _ = Sqlite.execute(db, "CREATE TABLE t (id INTEGER)", [])?
  let x = Sqlite.execute(db, "INSERT INTO t VALUES (1)", [])?
  println(Int.to_string(x))
  Sqlite.close(db)
  Ok(0)
end

fn main() do
  case run() do
    Ok(_) -> println("ok")
    Err(e) -> println(e)
  end
end
```

Verify this FAILS to compile with E0003 before the fix. Use the standard `compile_and_run` helper from e2e.rs.

**Step 2 — Diagnose the root cause:**

The E0003 (ArityMismatch) is emitted from `crates/mesh-typeck/src/unify.rs` in three locations:
- Line 292: `Ty::Fun` parameter count mismatch
- Line 333: `Ty::App` type argument count mismatch
- Line 360: `Ty::Tuple` element count mismatch

Add temporary `eprintln!` debugging BEFORE each of these three error sites to print the types being unified when ArityMismatch fires. Run the reproduction test and identify WHICH of the three branches fires and WHAT the two types are.

The hypothesis is that during `infer_call` (line 4506-4619), the `ret_var` fresh variable is returned unresolved at line 4619 (`Ok(ret_var)`). When the try operator (`infer_try_expr`) processes the operand, it resolves the type and extracts `ok_ty`. However, the `ok_ty` might still be a type variable that later gets unified with a function type or application type during constraint propagation, causing an arity clash.

Alternatively, the bug may involve the `Ty::Var` branch in `infer_try_expr` (lines 7925-7943): when the call's return type variable hasn't been resolved yet by the time the try-expression processes it, the code creates ADDITIONAL fresh variables and unifies the operand with `Result<fresh_t, fresh_e>`. This creates a second set of Result type variables that compete with the original unification from `infer_call`, potentially creating inconsistent constraints.

**Step 3 — Fix based on diagnosis:**

The likely fix is one of:
1. **In `infer_call` (line 4619)**: Return `Ok(ctx.resolve(ret_var))` instead of `Ok(ret_var)` so the try-expression receives a fully resolved type. This prevents the `Ty::Var` fallback branch in `infer_try_expr` from creating competing type variables.

2. **In `infer_try_expr` (line 7897-7898)**: After calling `infer_expr` for the operand, do a deeper resolution that fully resolves all nested type variables before matching on `resolved`. This might involve resolving TWICE or using a `fully_resolve` helper.

3. **In `infer_try_expr` (lines 7925-7943)**: When the `Ty::Var` branch fires, check if the variable is actually bound (just not yet resolved through the union-find chain) before creating new fresh variables.

Apply whichever fix the diagnosis reveals is correct. Remove the temporary debugging `eprintln!` statements after fixing.
