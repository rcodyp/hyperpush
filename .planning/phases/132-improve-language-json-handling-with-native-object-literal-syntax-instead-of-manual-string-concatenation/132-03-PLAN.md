<!-- JSON patterns to migrate. Before/after reference from CONTEXT.md: -->

Pattern 1 â€” Simple string literal with escaped quotes:
```
# Before:
"{\"error\":\"unauthorized\"}"
"{\"status\":\"accepted\"}"
"{\"type\":\"filters_updated\"}"
# After:
json { error: "unauthorized" }
json { status: "accepted" }
json { type: "filters_updated" }
```

Pattern 2 â€” String concatenation with variables:
```
# Before:
"{\"error\":\"" <> reason <> "\"}"
"[WS] Connection closed: " <> String.from(code)  # NOT a JSON pattern, skip
# After:
json { error: reason }
```

Pattern 3 â€” Heredoc interpolation:
```
# Before:
"""{"error":"#{reason}"}"""
"""{"type":"event","issue_id":"#{issue_id}","data":#{body}}"""
"""{"status":"ok","affected":#{n}}"""
# After:
json { error: reason }
json { type: "event", issue_id: issue_id, data: body }
json { status: "ok", affected: n }
```

Pattern 4 â€” Multi-field complex concat (in api/detail.mpl):
```
# Before:
"{\"id\":\"" <> id <> "\",\"project_id\":\"" <> project_id <> "\",\"level\":\"" <> level <> "\"}"
# After:
json { id: id, project_id: project_id, level: level }
```

IMPORTANT â€” What NOT to migrate (leave as-is):
- `"[WS] Connection closed: " <> String.from(code)` â€” not JSON, it's a log message
- `"project:" <> project_id` â€” not JSON, it's a room name string
- `"[" <> String.join(items, ",") <> "]"` â€” JSON array construction (json { } only does objects)
- Expressions that produce JSON arrays â€” json { } is object-only; arrays remain as-is
- `"{\"next_id\":" <> next_str <> ...}` where values are already JSON strings (may double-encode) â€” evaluate case by case
- `"{\"event\":" <> detail_json <> ...}` â€” detail_json is already a JSON string; nesting it in json { } would double-encode it as a quoted string. Leave these patterns ALONE or handle carefully.

CRITICAL: The fields `data`, `event`, `navigation` etc. in complex nested patterns where the VALUE is already a JSON-encoded string MUST NOT be naively converted â€” they would be double-encoded. For those, either:
a. Leave them as heredoc/concat (it's valid to leave complex cases), OR
b. Only migrate the simple single-level patterns

Recommended strategy: Migrate ALL patterns where values are bare Mesh variables (Int, String, Bool, nil) or String literals. Skip patterns where the value is already a pre-encoded JSON string being spliced in raw.

Add this section:
```markdown
## JSON Literals

Use `json { }` to construct JSON objects without manual string escaping or interpolation:

```mesh
# Simple object literal
let response = json { status: "ok", count: 42 }
# response is a String: {"status":"ok","count":42}

# Multi-line (same result)
let event = json {
  type: "event",
  issue_id: issue_id,
  severity: "high"
}
```

Keys are bare identifiers (no quotes needed). Values are any Mesh expression â€” the type determines how they are serialized:

| Mesh type | JSON output |
|-----------|-------------|
| `String`  | `"quoted string"` |
| `Int`     | `42` (unquoted number) |
| `Float`   | `3.14` (unquoted) |
| `Bool`    | `true` / `false` |
| `nil`     | `null` |
| `Option<T>` | `null` (None) or the value (Some) |

The result of `json { }` is a `String` and can be passed anywhere a `String` is expected â€” for example, directly to `HTTP.response` or `Ws.broadcast`:

```mesh
HTTP.response(200, json { status: "ok", affected: n })
HTTP.response(401, json { error: "unauthorized" })
Ws.broadcast(room, json { type: "update", id: record_id })
```

This replaces heredoc JSON templates (`"""{"key":"#{val}"}"""`) and manual string concatenation (`"{\"key\":\"" <> val <> "\"}"`) with readable, type-safe object literals.
```
